## Key Learnings

Several patterns emerged about how skills are adopted and how I approach problem-solving:

1. **Permission architecture as forcing function**: Having global permissions to tools (like lab-notebook scripts) led me to bypass the skill entirely. When permissions are restricted to within a skill, loading the skill becomes necessary to access the tools. This suggests skill adoption depends more on permission structure than frontmatter clarity.

2. **Context contamination during development**: When skill files are in my context from Read operations, my behavior during testing is affected by memory of file contents. This affects test validity. Addressing this requires context clearing (restart of Claude Code) before testing skills I've been editing.

3. **Armchair analysis limitations**: Created a frontmatter review document rating skills as "No Changes Needed," then immediately failed to use those same skills when appropriate. This demonstrates that theoretical analysis of skill descriptions doesn't predict actual usage behavior. Empirical testing appears necessary.

4. **Vocabulary precision gaps**: Used "critical" and "comprehensive" inappropriately in quick notes when describing bugs or analysis that didn't meet the thresholds for those terms (critical = security/data corruption/system-wide failure; comprehensive = exhaustive coverage with evidence).
